# RHODES v0.4.6 — Audits closed. Dashboard re-built.

Released **2026-05-14**.

v0.4.5 was the morning. v0.4.6 is the afternoon. Six branches merged in a single sequence: five close every HIGH-severity finding from the overnight audits except dashboard auth (queued for v0.4.7), one rebuilds the dashboard from the developer HUD you saw at lunchtime into the customer-grade React app the product deserves.

---

## Per-step approval gates actually gate.

The v0.4.5 approval gate keyed decisions by `plan_id` alone. The first plan-level approval auto-resolved every later per-step `requestApproval` against the same plan — so a multi-tier plan with a deliberate "approval checkpoint before the destructive snapshot delete" step would get its destructive gate silently auto-approved by the original plan-level decision. The 11-step esxi-01 save from 2026-05-13 was the live worked example.

v0.4.6 keys decisions by `(plan_id, step_id)`. A plan-level decision still resolves the plan-level entry; a separately-issued per-step gate creates a distinct pending entry that needs its own operator confirmation. The dashboard renders per-step gates as separate sibling cards with a visible "step \<id\>" badge; the v0.4.5 `?plan=<id>` deep-link gains `&step=<step_id>` refinement.

**What this means for you:** `policy.orchestration.approval.explicit_tiers` is no longer a paper rule. When the agent's plan includes a tier-2 step and a later destructive step, you sign off on each one — not once, generally, at the top.

## LLM calls don't wedge the agent anymore.

A hung Anthropic/OpenAI call could wedge `agentCore.run` for up to the SDK default (~10 min), burning a `maxConcurrentHeals` slot and blocking other incidents. v0.4.6 puts every LLM call inside an `AbortController` with configurable timeouts (60s plan-step, 30s inline-reasoning step) and a retry-once semantic on timeout. A new `AgentEventType.LlmTimeout` event surfaces in the audit log and the Telegram alert bridge.

**What this means for you:** RHODES degrades gracefully when the LLM provider stalls. The slot frees up, other incidents keep getting handled, the operator sees a clear "LLM call timed out after Xms" failure rather than a phantom hang.

## ChaosEngine respects its own approval gate.

Security audit X-1: `ChaosEngine.execute` checked `requires_approval && risk_score > 70` but only mutated a status string — execution proceeded regardless. v0.4.6 wires the actual approval gate: when the threshold trips, `ChaosEngine` `await`s `approvalGate.requestApproval(...)` and only proceeds on approve. Five new chaos event types (`ChaosApproved`, `ChaosRejected`, `ChaosApprovalTimeout`, `ChaosBlocked`, `ChaosAudited`) flow into the Telegram alert bridge.

Plus a NEVER list — `vm_destroy`, `delete_volume`, `wipe_node`, `format_storage`, and anything whose id / action type / action description matches `/\b(destroy|delete|wipe|format)\b/i` — that's unconditionally blocked. No approval option, period.

**What this means for you:** `vm_kill` and `node_isolate` chaos scenarios still need your explicit approval. `vm_destroy` doesn't ask — it's not on the menu. The chaos engine is now bounded both ways: opt-in for risk, opt-out hardcoded for irreversibility.

## `configureService` is no longer a command-injection surface.

Security audit C-1: the old path interpolated `service` and `config_path` directly into `systemctl ${service}` and `cat > '${configPath}'` shell strings, bypassing the SSH safety classifier. v0.4.6 hardens it with strict input validation (regex allowlists for service names, prefix allowlists for config paths covering `/etc/`, `/var/lib/`, `/usr/local/etc/`, `/opt/`, `/srv/`, null-byte and `..` rejection) plus a classifier re-check per step. File writes go through `tee` stdin-fed instead of `cat > ...` shell redirection.

Side benefit: the SSH safety classifier's `systemctl-mutate` rule was widened from `(restart|reload|stop|start)` to `(restart|reload|stop|start|enable|disable|mask|unmask|daemon-reload)` — those verbs were previously falling through to the fail-closed destructive default, which over-blocked legitimate `systemctl enable nginx` calls.

**What this means for you:** even when the LLM gets creative with service names — and it does — the bumper catches anything outside the regex. `nginx; rm -rf /` was a valid old-path input. It's a 400 now.

## Trigger collisions no longer lose playbooks.

The v0.4.4 release notes promised that when an HTTP probe fails for a service like Jellyfin, two playbooks fire: `jellyfin-service-probe` (simple service restart) and `vm_in_guest_diagnostic` (full SSH-in investigation). That was a lie — the healing engine was taking `playbookEngine.match(anomaly)[0]` and silently dropping every other match. v0.4.6 emits ALL matching playbooks. Each respects its own cooldown / approval / max_retries; `maxConcurrentHeals` still bounds the in-flight count.

**What this means for you:** when Jellyfin actually goes 503, both the restart attempt AND the in-VM diagnostic plan land in parallel. Restart usually wins; when it doesn't, the diagnostic plan is already there with the SSH plan attached.

## The dashboard is the dashboard now.

The page you saw earlier today was the developer HUD — `src/frontends/dashboard/template.ts`, a single TypeScript-generated HTML file with inline SVGs. That was the fallback that served while `dashboard-v2/dist/index.html` didn't exist on the NUC. v0.4.6's deploy step builds `dashboard-v2/` on the NUC, and the React + Vite + Zustand customer-grade app renders on first paint instead.

What changed in dashboard-v2 between the May 12 build and v0.4.6:

- **Real header** — RHODES lockup top-left, mono uppercase tagline *"Infrastructure, executed."* underneath, right-side row of six small mono rectangles (version, shadow-mode indicator with pulsing dot, uptime, providers connected, sse_clients count, LIVE/DOWN pip) wired to `/api/healthz` + `/api/cluster/summary`.
- **Pending Approvals panel** — floating panel above page content, hidden when empty. Cards render Approve/Reject buttons + expandable agent reasoning + mono-rect tier badges. Safety snapshots from the v0.4.3 retention rule get a 🛡️ badge and a green border so they look distinctly non-destructive. Live via SSE + 10s catch-up poll.
- **Remediate button** on open/healing incidents — clicks template a natural-language prompt from incident labels and `POST` to `/api/agent/command`. Same four mapping branches the legacy HUD had.
- **Deep-link handler** — `?plan=<id>` (optional `&step=<step_id>`) scrolls the matching card into view, pulses Rhodes Blue for 2s, falls back to a toast if the plan was already decided.
- **Sidebar regrouped** into Operations / Infrastructure / Insights with underline-accent active state. No more left-edge stripes.
- **Playbooks page rebuilt** — 9 playbooks with per-trigger icons + colors.

And the operator-flagged "weird blue lines" (you specifically called them out) are gone:

1. `.ov-stat-accent` — 3px top accent stripes on Overview status cards: hidden.
2. `.ov-provider-row` — colored left-edge stripes per provider: replaced with neutral border.
3. `.prediction-card.status-*` + `.rightsizing-card.severity-*` — 3px colored `border-left` stripes: normalized.
4. `.pipe-step.running .pipe-step-line` — bright-blue vertical gradient on the active plan list: softened from 100% to 50% Rhodes Blue.

Empty states get the RHODES mark at 18% opacity above 14px steel-text copy. *"No active incidents — RHODES is watching."*
