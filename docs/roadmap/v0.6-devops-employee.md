# v0.6 — RHODES as DevOps Employee

**Status:** scoping draft, 2026-05-15
**Owner:** Pranav (Sher Systems)
**Successor to:** v0.5.0 (Slack-as-SRE-surface)

## Thesis

v0.5.0 made RHODES a smart notifier + auto-restarter. v0.6 makes it the
DevOps employee a customer can stop hiring: it triages with shell
access, upgrades vSphere clusters, and treats every hypervisor in the
fleet the same way a senior SRE would. The pitch reframes from
"observability + auto-remediation for one VM at a time" to "headcount
replacement for the boring 80% of infra work."

This is what the "AI brain for infra / Claude Code for infrastructure"
positioning has been building toward. v0.5.0 proved the autonomy
loop on one trivial action (restart). v0.6 stress-tests that loop on
the hardest action a VMware admin runs all year: cluster upgrades.

## Anchor demo for the v0.6 release

A customer files a ticket: *"upgrade my vSphere cluster prod-east from
8.0u2 to 8.0u3."*

1. RHODES generates an upgrade plan, posts it as `approval_needed`
   Block Kit in `#rhodes-approvals` with: target hosts, baseline image,
   pre-flight check matrix, estimated duration, evacuation plan,
   rollback plan. Approve/Reject buttons.
2. Operator approves.
3. RHODES executes the plan rolling, host-by-host, posting per-step
   progress as **thread replies** under the approval message:
   - Pre-flight: vCenter version compat, DRS automation level, host
     hardware compat, capacity check (can the remaining hosts hold
     evacuated workload?), open alerts gate.
   - Per host: enter maintenance → DRS evacuates → remediate via LCM →
     wait for reboot → exit maintenance → smoke-test (1 VM round-trip)
     → next host.
   - Auto-rollback ladder on any failure: re-image host from previous
     baseline if remediation step failed; abort upgrade and DM the
     operator if the rollback itself fails.
4. On completion: a `ticket_resolved`-shaped DM with an LLM postmortem
   ("upgraded 4/4 hosts, total wall-clock 1h 12m, no VM downtime
   thanks to vMotion, 2 ESXi hosts took an extra reboot cycle for
   firmware staging").

If any single host fails twice, RHODES halts the cluster upgrade,
rolls back the affected host to the previous baseline, posts the
failure with the diagnostic bundle (logs + esxcli output via the
SSH adapter), and waits for an operator decision.

This is the demo Pranav records for the v0.6 release video and for
the next investor pitch — it's literally "you watch a video and a
cluster upgrades itself."

### v0.6.0 — Maintenance mode + DRS primitives (1–2 weeks)

The orchestrator's atomic operations. Without these, the upgrade flow
can't even put a host into maintenance mode.

**Code:**
- `src/providers/vmware/client.ts` — extend with:
  - `enterMaintenanceMode(host, { evacuate: bool, timeoutSec, drsAutomationLevel })`
  - `exitMaintenanceMode(host)`
  - `getMaintenanceModeStatus(host)`
  - `triggerDrsRebalance(cluster)`
  - `getDrsRecommendations(cluster)`
- `src/providers/vmware/adapter.ts` — new tools:
  - `vmware_host_enter_maintenance` (tier: `risky_write`)
  - `vmware_host_exit_maintenance` (tier: `safe_write`)
  - `vmware_host_get_maintenance_status` (tier: `read`)
  - `vmware_drs_recommendations` (tier: `read`)
  - `vmware_drs_apply` (tier: `risky_write`)
- `src/providers/vmware/types.ts` — `HostMaintenanceStatus`, `DrsRecommendation`

**Tests:**
- `tests/providers/vmware-maintenance.test.ts` — mock vSphere REST, verify enter/exit round-trip
- Governance alignment check: enter-maintenance ≥ `risky_write`

**Decision needed from Pranav:** Do we require DRS-fully-automated as
a precondition for maintenance-mode-with-evacuate, or do we surface
DRS-recommendations and ask the operator to approve the evacuation
plan? **My take:** the latter — gives the operator visibility into
*what* moves where before the upgrade starts, and a kill switch.

### v0.6.5 — Snapshot operations completed + pre-flight check engine (1 week)

**Code:**
- Finish `vmware_create_snapshot` / `vmware_delete_snapshot` /
  `vmware_revert_snapshot` (the SOAP-only ones marked "not yet
  implemented" in the adapter). These are the rollback substrate.
- `src/healing/preflight.ts` — new module:
  - `PreflightCheck` interface: `{ id, description, severity, run(ctx) → PASS/WARN/FAIL }`
  - `PreflightReport` aggregating check results
  - `runPreflight(plan, checks[])` → all-or-nothing decision
- A small library of vSphere-specific checks:
  - `host-compat-matrix-check` — target version supports host CPU
    family, host model is on HCL.
  - `cluster-capacity-check` — N-1 capacity, can absorb evacuation
    of any single host.
  - `drs-automation-level-check` — DRS must be at least
    "partially-automated".
  - `open-alerts-gate` — abort if any non-info alert is firing on a
    target host (no upgrading during an active incident).
  - `vcenter-version-precedes-host-target` — vCenter must be
    upgraded to ≥ host target version *before* any host upgrade.

**Tests:**
- `tests/healing/preflight.test.ts` — fixture-based per-check verification.
- Snapshot revert round-trip in `tests/providers/vmware-snapshots.test.ts`.

### v0.7.0 — Cluster Upgrade Orchestrator (2–3 weeks, the headline)

**Code:**
- `src/healing/upgrade-orchestrator.ts` — new top-level module. The
  state machine for cluster upgrades. Reuses `IncidentCoordinator`'s
  `resolveIncident` pattern (one bundled call that fires hooks +
  ticket sync) — every state transition emits an event and a Slack
  thread update.
- vSphere LCM integration in `src/providers/vmware/client.ts`:
  - `getClusterDesiredState(cluster)`
  - `setClusterDesiredState(cluster, image)` (the LCM "image-based"
    target spec)
  - `remediateHost(host)` — the LCM "remediate" verb (apply pending
    image to one host)
  - `getRemediationStatus(host)`
  - `validateClusterDesiredState(cluster, image)` (LCM pre-flight)
- Adapter tools:
  - `vmware_cluster_get_desired_state` (`read`)
  - `vmware_cluster_set_desired_state` (`destructive` — cluster-scope)
  - `vmware_host_remediate` (`destructive`)
  - `vmware_host_remediation_status` (`read`)
- `src/healing/rollback.ts` — a "rollback ladder" type that pairs
  every upgrade-time mutation with its inverse. The orchestrator
  pushes each successful mutation onto a stack at execution time; on
  failure it pops and applies inverses in reverse order. Inverse
  primitives include: `revert-snapshot`, `re-image-to-previous`,
  `exit-maintenance`, `set-drs-original-level`.

**State machine (rough sketch):**

```
PROPOSED ─approve─▶ PREFLIGHT ─pass─▶ PER_HOST_LOOP
                       │                  │
                       └─fail─▶ ABORTED   │
                                          ▼
                                  enter_maintenance
                                          │
                                          ▼
                                  drs_evacuate
                                          │
                                          ▼
                                  remediate_host ───fail─▶ ROLLBACK_HOST
                                          │                     │
                                          ▼                     ▼
                                  await_reboot          exit_maintenance
                                          │                     │
                                          ▼                     ▼
                                  exit_maintenance      DM_OPERATOR_AND_HALT
                                          │
                                          ▼
                                  smoke_test
                                          │
                                  ──fail──┴──pass──▶ next host / DONE
```

**Tests:**
- `tests/healing/upgrade-orchestrator.test.ts` — state-machine
  transitions in isolation with mocked vSphere client.
- `tests/healing/rollback.test.ts` — ladder pops on each failure
  type, doesn't double-pop, handles inverse-failure.

**Approval & governance:**
- Cluster-scope mutations (`vmware_cluster_set_desired_state`,
  `vmware_host_remediate`) are tier `destructive`. They require
  approval at plan-creation time, NOT per-host (would otherwise spam
  the operator with 4× approval requests per cluster).
- The plan is approved once. Each per-host action is gated by a
  pre-host pre-flight check that can abort the plan but does not
  require fresh approval. This matches how a human SRE works —
  approve the cluster upgrade once, watch progress, intervene only
  if something goes sideways.

**Decision needed from Pranav:** Do we support "vCenter upgrade
first, then host upgrades" as a single coordinated plan, or treat
vCenter upgrade as a separate (manual or RHODES-orchestrated) phase?
**My take:** separate phase for v0.7. vCenter upgrade has different
failure modes (downtime, certificate cycling, DB upgrade) and
deserves its own orchestrator. Tackle in v0.7.5 if v0.7.0 lands clean.

### v0.7.5 — Multi-hypervisor parity (1–2 weeks)

The AWS / Azure / Kubernetes adapters are registered but don't have
the autopilot rules and playbooks that Proxmox has. Bring them up
to parity for the same restart/healing flows v0.5.0 demonstrated:

- AWS EC2: detect stopped instance → auto-restart → DM operator. Same
  flow we have for Proxmox VMs.
- Azure VM: same.
- Kubernetes: pod restart on crash-loop, deployment rollback on bad
  rollout.

This isn't about NEW capabilities — it's about closing the
"works-on-VMware-and-Proxmox but-not-cloud" gap so the multi-hypervisor
positioning has teeth.

**Decision needed from Pranav:** Cloud autopilot rules need cloud
credentials in production environments. Sher Systems' demo
environment (NUC + Proxmox) doesn't have an AWS account RHODES can
actually act in. Two options:
1. Use the existing $4.50/mo canary (per `feedback_aws_spend_guardrail.md`)
   for read-only validation only — defer write tests until customer.
2. Spin up a sandbox AWS account for CI tests, isolated from
   production.

**My take:** option 1. Customer-driven write actions when there's a
customer. Don't burn money on what nobody's paying for yet.

### v0.8.0 — vCenter upgrade orchestration (2 weeks)

If v0.7 lands, the natural follow-on. vCenter upgrades have different
shape (single VM, but with appliance-level DB migration, cert

Out of scope for this scope doc; revisit after v0.7 ships.

## Cross-cutting concerns

### Action tiering for upgrades

The existing 5-tier governance (`read`/`safe_write`/`risky_write`/`destructive`/`never`)
works for most things but is undersized for upgrade plans, which are
*coordinated sequences of destructive actions across multiple hosts*.
We don't need a new tier — we need a **plan-level approval** that
authorizes a sequence, with per-step pre-flight gates.

Proposal: extend `src/governance/policies.ts` with a `plan_approval`
governance hook that fires once at plan creation. Per-step
governance still runs but uses the plan_approval as a credential —
"this step is part of approved plan X." Audit trail records both the
plan approval and the per-step gate.

### Maintenance/silence mode (the v0.5.1 RCA-hallucination fix)

The "operator stopped this VM" → "RHODES thinks it's a crash" problem
gets WORSE when upgrades start, because every upgrade puts a host
into maintenance mode and evacuates VMs — RHODES would see each
evacuated VM as "missing" and try to remediate.

Build `src/healing/silence.ts`:
- `silenceWindow(target, durationSec, reason)` — suppresses
  incident creation for matching targets.
- The upgrade orchestrator opens a silence window covering the
  cluster for the upgrade duration before it starts.
- Manual operator silences via Slack: `/rhodes silence vmid 101 for 1h reason "scheduled reboot"`.

This is a v0.5.1 gap that v0.6 hard requires.

### Observability of long-running plans

A cluster upgrade can take an hour. The current dashboard model is
incident-centric — there's no "active plan progress" view that
visualizes a multi-step orchestration.

Either:
- Extend the existing `Active Plan` tab (already present in
  `dashboard-v2/src/components/ActivePlan.tsx`) to render upgrade
  state-machine progress.
- Or add a dedicated `Upgrades` tab.

**My take:** extend `Active Plan` — keeps the dashboard simpler.

### Backwards compatibility / migration story for customers

If RHODES is deployed at a customer with vCenter 7.x and the upgrade
orchestrator only supports vCenter 8.x LCM image-based remediation,
that customer can't use the headline feature. The LCM API surface
changed substantially across versions.

**Decision needed from Pranav:** Support range — vCenter 7.0u3+ only?
**My take:** yes. vCenter 7.0u3 introduced image-based LCM; earlier
versions used baseline-based vUM which has different APIs and is
deprecated. Stake the feature on the modern API; document the floor.

## Risks and how we mitigate

| Risk                                                       | Mitigation                                                                                                                      |
| ---------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| Upgrade fails mid-flight, cluster in inconsistent state    | Rollback ladder runs reverse mutations; if rollback fails, halt + DM operator + bundle diagnostics                              |
| LCM API rate limits during cluster-wide remediation        | Sequential per-host, no parallelism (matches vSphere best practice anyway)                                                      |
| RHODES misidentifies a VM as crashed during upgrade        | Silence window opened by orchestrator before upgrade starts; covers entire cluster + duration                                   |
| Operator approves plan, walks away, plan stalls            | Plan timeout (default 4h) → halt + DM operator. No silent stalls.                                                              |
| Live customer hits us with vCenter < 7.0u3                 | Floor is documented; pre-flight check rejects with a clear "upgrade vCenter first" message; manual baseline-based path is a separate feature if customers actually need it |
| LCM "remediate" succeeds but host is in degraded state     | Post-remediation smoke test (1 VM round-trip) catches this; halt cluster, DM operator                                          |
| Authentication: vSphere SSO with AD-backed accounts        | Service account approach documented in `provider-authoring-guide.md`; credentials in `.env`, rotation via standard process    |

## Out of scope for v0.6

- NSX-T / NSX-V upgrades — separate orchestrator, separate cycle.
- vSAN upgrades — included implicitly via host firmware in LCM image,
  but no vSAN-specific health gates beyond what LCM ships with.
- Aria Operations / vROps integration — out of scope; LCM provides
  enough signal.
- Windows guest patching / linux distro upgrades inside VMs —
  separate problem space.
- Migration of workloads between hypervisors during upgrade —
  already covered by the existing migration subsystem
  (`docs/architecture.md` §6); not part of this scope.

## What's NOT changing

- The Slack DM thread surface from v0.5.0 stays the operator's main
  control point. Upgrade progress lands as thread replies under the
  approval message, same shape as the ticket-resolved flow.
- The five-tier governance model is the base; we extend it with
  plan-level approval, we don't replace it.
- The provider adapter pattern stays — every new vSphere capability
  goes through `src/providers/vmware/`, never around it.
- v0.5.0 invariants (autonomous restart, no approval for trivial
  recovery, postmortem on resolve) keep working as-is.
